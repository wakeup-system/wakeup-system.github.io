Requirements

The WakeUp System's Requirements.

This page outlines the essential functional and non-functional requirements crucial for the development, implementation, evaluation, and long-term sustainability of the Wakeup System.

Functional Requirements

  • Patient Monitoring and Intraction via Triggers - The system should be able to monitor patients through a computer vision algorithm to interpret the patient's body/face movements and a sound/voice detection algorithm to recognise patterns created by the patient through various forms (e.g.: mouth noises, eye blinking, etc.). This requirement will allow patients with disabilities to interact with smart devices in the easiest way possible.
  • Ease of System Interaction - It is important that the patients can initiate an interaction with the system easily and be reminded consistently that the system exists for interacting with smart home devices around them.
  • Graphical User Interface for Medical Staff - The GUI should allow the medical staff or patients to easily configure custom commands for the triggers, as well as mapping the triggers to target devices, enabling them to use the system in the simplest way possible.
  • Usage Tracking and Engagement Insights - The system should be able track its usage over time. The Medical staff will have the ability to assess the patient’s interaction with the system. This information will help determine the frequency and preference for its use, providing valuable insights into the patient’s engagement with the system.

Non-Functional Requirements

  • Response time - The system should provide immediate feedback to patient interactions, with a maximum response time of 2 seconds for trigger detection and initiating actions on target devices. Quick response times are critical in healthcare settings, especially in emergency situations or for patients with disabilities, to reduce discomfort and potential harm.
  • Scalability - The system should be easily scalable to support an increasing number of triggers and targets without significant performance decrease. Importantly, adding new triggers and targets should be managed easily without the need for complex configurations. Since this project aims to lay the foundation of a project continued by other teams in the future, it is important that the system is modular, easy to maintain and extendable.
  • Security - We plan to ensure that the security of the system is designed to protect patient confidentiality rigorously, limiting the use of patient data to only name and patient ID in compliance with healthcare regulations. This approach minimises the risk of data breaches while ensuring the system adheres to privacy standards.
  • Ease of installation - The installation process should be streamlined for efficiency, with each trigger being managed through executable files. This design choice simplifies the setup process, enabling medical staff to easily integrate new components into the system without complex configurations, facilitating rapid deployment in a healthcare environment.
  • Integration with existing healthcare systems and standards - The system should be designed to integrate seamlessly with existing healthcare systems and standards, such as FHIR (Fast Healthcare Interoperability Resources) for data exchange (e.g., statistical usage data of the system).
  • Open Source Project - The software should be an open source project to ensure continued accessibility for future students or developers who may need access to the source code for future projects.
